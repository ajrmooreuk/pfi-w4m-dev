# ARCH-CICD-003: Glossary — CI/CD & Multi-Instance Terms

**Version:** 1.2.0
**Date:** 2026-03-17 (updated — role names formalised: Strategy Analysis (VE), Security Architect)

---

## Platform Architecture Terms

| Term | Abbreviation | Definition |
|------|:------------:|------------|
| **Platform Foundation Core** | PFC | The shared foundation of ontologies, tools, agents, and design system assets maintained in the Azlan-EA-AAA repository. Read-only for instances. |
| **Platform Foundation Instance** | PFI | A client-specific or domain-specific deployment that consumes PFC-Core assets and adds instance-specific data, configuration, and tooling. |
| **Core Forge** | -- | The Azlan-EA-AAA repository in its role as the origin of all PFC assets. Ontologies are authored, tested, and released here. |
| **Instance Triad** | -- | The three repositories (dev/test/prod) that form a PFI's promotion pipeline. Each triad is independent. |
| **Convention Manifest** | -- | `convention-manifest.json` — the list of files that are promoted across repos and synced to live instances. |
| **Live Repo** | -- | A downstream repository registered in `live-repos.json` that receives convention and core asset updates from prod. |
| **Version Pin** | -- | The `azlan-workflow-version` file in each live repo that controls which PFC release version it accepts. Set to `latest` for auto-sync or a specific tag (e.g., `pfc-v1.0.0`) to freeze. |

## Instance Ownership Terms

| Term | Definition |
|------|------------|
| **OWNER_INSTANCE** | A top-level PFI with its own triad that owns the products and client instances beneath it (e.g. W4M-RCS, AIRL, BAIV). |
| **PRODUCT** | A distinct product within an owner PFI, with its own project boards (e.g. AZA under W4M-RCS). May share the owner triad or have its own. |
| **CLIENT_SUB_INSTANCE** | A private deployment of a product for a specific client, living under the owner PFI's brand or as a named sub-instance. |
| **CLIENT_PFI** | A full PFI triad provisioned exclusively for a client, owned and governed by a Partner PFI, operating under the partner's brand. Client owns their data. |
| **Owner PFI** | The `OWNER_INSTANCE` that governs (administers Engineering project, holds `parent_pfi` reference for) all products and client instances beneath it. |
| **parent_pfi** | Registry field identifying the `OWNER_INSTANCE` that governs a `PRODUCT`, `CLIENT_SUB_INSTANCE`, or `CLIENT_PFI`. Null for `OWNER_INSTANCE` entries. |
| **instance_type** | Registry field classifying each entry as `OWNER_INSTANCE`, `PRODUCT`, `CLIENT_SUB_INSTANCE`, or `CLIENT_PFI`. Drives permission defaults and backup tier. |

## GitHub PM Automation Terms

| Term | Definition |
|------|------------|
| **Engineering project** | The GitHub Projects v2 board for a PFI dev repo that tracks instance infrastructure work (CI/CD, scaffold, devops). Hub admin = Admin; all collaborators = Read. |
| **Product project** | The GitHub Projects v2 board for a PFI dev repo that tracks product work (epics, features, stories, RAID, briefs). Collaborators = Write. |
| **Two-project model** | The standard configuration of one Engineering project and one Product project per PFI dev repo, applied at every hierarchy tier. |
| **PROJECT_PAT** | A GitHub Personal Access Token with `project` scope, held exclusively by Hub admins. Required to create projects and for `auto-add-to-projects.yml` to route issues. |
| **auto-add routing** | Label-driven workflow (`auto-add-to-projects.yml`) that automatically adds new issues to Engineering or Product project based on `type:` and `domain:` labels. Generated by `setup-repo` skill at instance creation. |
| **workflow_call stub** | A thin PFI-repo workflow file that delegates execution to the hub's canonical workflow via `workflow_call`. Eliminates replication drift. |
| **engineering_project_id** | Registry field — the `PVT_...` GitHub Projects v2 node ID for the Engineering project. Written by `setup-repo` skill. |
| **product_project_id** | Registry field — the `PVT_...` GitHub Projects v2 node ID for the Product project. Written by `setup-repo` skill. |
| **PVT node ID** | The globally unique identifier assigned by GitHub to every Projects v2 board, prefixed `PVT_`. Used in GraphQL API calls to read/write project fields and items. Not visible in the GitHub UI — obtained via `gh api graphql`. Required by `auto-add-to-projects.yml` and `set-pbs-id.yml` to route issues to the correct board. |

## Backup & Resilience Terms

| Term | Definition |
|------|------------|
| **backup_tier** | Registry field classifying each repo's backup priority. Derived from `instance_type` — never set manually. T0 = Hub/PFC triad (critical); T1 = all PFI and client instances (material). |
| **T0** | Backup tier for Hub (Azlan-EA-AAA) and PFC triad (pfc-dev/test/prod). Highest priority — platform cannot function without these. |
| **T1** | Backup tier for all PFI instances at any hierarchy level, including client-specific PFIs. Client data = T1 obligation regardless of instance size. |
| **backup auto-enrolment** | The mechanism (F61.24) by which adding a repo to the instance registry with a `backup_tier` automatically includes it in all backup scripts. No manual list maintenance. |

## CI/CD Pipeline Terms

| Term | Definition |
|------|------------|
| **Promotion** | The controlled transfer of convention files from one tier to the next (dev-to-test, test-to-prod) via PR. |
| **Sync** | The automated distribution of convention files from prod to all registered live repos when a new tag is pushed. |
| **Drift** | When a live repo's convention files diverge from the prod source of truth. Detected weekly by `drift-detection.yml`. |
| **Bootstrap** | One-command provisioning of a new repository with all conventions, workflows, labels, branch protection, and project board. |
| **Convention** | A standardised file (issue template, workflow, label definition, script) that is shared across all repos in the ecosystem. |
| **SME Approval** | Subject Matter Expert review required for test-to-prod promotions. Ensures quality gate before distribution. |

## Repository Architecture Terms

| Term | Definition |
|------|------------|
| **Three-Repo Model** | Using separate repositories (dev/test/prod) instead of branches for environment isolation. Rationale: branches share secrets, CI quota, and webhooks; repos share nothing. |
| **Hub-and-Spoke** | Architecture pattern where a central hub (PFC-Core) distributes assets to multiple spokes (PFI instances). |
| **Tier 0** | Architecture tier — Core Forge (Azlan-EA-AAA) and convention template repos. Also the backup tier label for Hub and PFC triad repos (see Backup & Resilience Terms). |
| **Tier 1** | Architecture tier — Convention pipeline (azlan-workflow-dev/test/prod). Also the backup tier label for all PFI instances (see Backup & Resilience Terms). |
| **Tier 2** | Architecture tier — Instance tier: PFI repo triads (pfi-{name}-dev/test/prod). |

## Ontology Distribution Terms

| Term | Definition |
|------|------------|
| **Ontology Library** | The `ont-registry-index.json` and all series directories containing OAA v6.1 compliant ontology files. Maintained in PFC-Core. |
| **Instance Data** | Instance-specific ontology instances (`*-instance-v*.jsonld`) that extend core ontologies with client/domain data. Created and maintained in PFI repos. |
| **Registry Snapshot** | A cached copy of `ont-registry-index.json` stored in each PFI repo at a pinned version. Updated on PFC release sync. |
| **GitHub Pages Deploy** | Static deployment of ontology library and visualiser. Provides URL-based ontology loading for all instances. |

## Design System Terms

| Term | Definition |
|------|------------|
| **Token Taxonomy** | The DS-ONT-defined hierarchy of design tokens (colour, typography, spacing, elevation) that forms the core design system. |
| **Brand Tokens** | Instance-specific token values (e.g., BAIV brand colours, W4M typography) derived from Figma extraction. |
| **Brand Switching** | Environment-variable-driven mechanism to swap active brand tokens without code changes. |
| **DS-ONT** | Design System Ontology — OAA-compliant ontology defining token types, component rules, and brand relationships. |

## Supabase Terms

| Term | Definition |
|------|------------|
| **Per-Instance Project** | Each PFI gets its own Supabase project (free tier). Provides full isolation of auth, database, and storage. |
| **RLS** | Row-Level Security — Postgres-native access control. Each PFI project scopes data access via RLS policies. |
| **Schema Template** | Shared database schema (5 core tables, 4 roles from RRR-ONT) that is applied identically to each Supabase project via migrations. |
| **Migration** | SQL file in `supabase/migrations/` that evolves the database schema. Applied via `supabase db push`. |

## Agent Terms

| Term | Definition |
|------|------------|
| **OAA** | Ontology Architect Agent — the AI agent governance framework (currently v6.1) that defines how ontologies are authored, validated, and composed. |
| **Claude Code Skills** | Slash commands (`/create-epic`, `/create-feature`, `/create-story`, `/review-hierarchy`, `/setup-repo`, `/setup-project-board`) packaged as a Claude Code plugin. |
| **Agent Prompt** | System prompt that configures an AI agent's behaviour, domain knowledge, and tool access. Maintained in PFC-Core and distributed to instances. |

## Platform Roles

| Role | Code | Full Name | Scope |
|------|------|-----------|-------|
| **Strategy Analysis** | VE | Strategy Analysis (VE) | Discovers, scopes, and models value opportunities. Owns VSOM, OKR, QVF, VP, Kano, PMF. Primary consumer of platform capabilities. |
| **Design Director** | DD | Design Director (DD) | Owns the design system, component architecture, and all visual/interactive output. Builds via DSY skill chain and App Skeleton. |
| **Process Engineer** | PE | Process Engineer (PE) | Owns technical architecture, integration contracts, delivery pipeline, and PPM. Arbitrates RBAC boundary conflicts. |
| **Context Architect** | FDN | Context Architect (FDN) | Owns infrastructure, security context, data sovereignty, and adaptive onboarding. Provides platform-layer context to all other roles. |
| **Compliance Authority** | CA | Compliance Authority (CA) | Owns compliance surface detection, audit boundaries, and regulatory requirements (GRC, DSPT, CAF, GDPR). Read-only on security surfaces — cannot sign SEC-DT-06. |
| **Security Architect** | SecArch | Security Architect | Owns security architecture decisions (SA-DT), threat model sign-off, and SEC-DT-06 dual sign-off authority. Independent of the 5 core roles — distinct authority for security gate decisions. Required for promote-to-prod (tdd-gate.yml G7). |
| **TDD Expert** | TDD-E | TDD Expert | HITL authority over the TDD-DT pipeline. Signs TDD-DT-07 gate and promote-to-prod PRs jointly with Security Architect. Eligible: PE or CA standing minimum. Cannot be automated or bypassed. |

> **Note on naming:** The VE role is formally **Strategy Analysis (VE)** — not "Value Engineer". The code `VE` is retained for brevity in filenames, skill IDs, and registry entries. All role-facing documentation must use the full name.

## Licence Terms

| Term | Definition |
|------|------------|
| **PFC-Standard** | Base licence tier — access to Foundation, one VE/PE series, and core tools. |
| **PFC-Compliance** | Compliance-focused licence — adds RCSG-Series ontologies and audit engine extensions. |
| **PFC-Enterprise** | Full licence — access to all ontology series, agent prompts, and design system core. |
| **Licence Control** | Enforced via version pinning + `live-repos.json` ontology scoping. Instances only receive ontologies listed in their licence scope. |
