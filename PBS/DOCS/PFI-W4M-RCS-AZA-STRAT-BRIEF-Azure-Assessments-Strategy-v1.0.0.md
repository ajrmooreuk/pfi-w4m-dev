## Executive Summary

This brief evaluates how PFI-AIRL can strategically leverage **Microsoft Azure Skills Builder** and **Azure Assessment tooling** (Well-Architected Review, Cloud Adoption Framework, Azure Migrate) as upstream inputs to its AI-augmented assessment offering.

The strategic finding is clear: **Microsoft's assessment tools create the demand signal; PFI-AIRL captures and converts it.** Azure assessments surface organisational gaps but produce no executable sprint roadmap. Azure Skills Builder identifies team capability deficits but does not connect them to AI project readiness. PFI-AIRL's ontology-driven, PbD/SbD-gated execution model fills exactly that gap — and does so with capabilities that Microsoft cannot replicate within its commercial model.

The **Insurance Snapshot Audits** — rapid structured assessments originally developed for the insurance sector proposal pipeline — are the operational prototype for this model. They demonstrate that structured, time-bounded discovery produces actionable findings faster than any framework-led audit. Azure Assessments are, in effect, a Microsoft-branded Snapshot Audit at platform scale. The strategic move is to treat Microsoft assessment outputs as structured Snapshot Audit inputs, and run the AIRL execution layer on top.

**Three immediate strategic moves emerge from this analysis:**

1. Build the WAR/CAF → AIRL ingest pipeline (agentic, <2 hours to sprint plan)
2. Map Azure Skills Builder gaps to AIRL 9-domain readiness scores (team-to-project bridge)
3. Register PFI-AIRL in Microsoft partner co-sell programmes as the post-assessment execution partner

## 1. Context: The Insurance Snapshot Audit as the Prototype

### 1.1 What the Insurance Snapshot Audits Established

The Insurance Snapshot Audits — developed as part of the AIRL pipeline for insurance sector prospects — demonstrated a critical commercial pattern:

> **A structured, time-bounded assessment (≤5 days) delivered more strategic clarity than a 60-day framework audit, because it was scoped to produce decisions not documentation.**

The Snapshot Audit format:

- Defined discovery questions mapped to risk/readiness domains
- Produced a scored maturity profile per domain
- Generated a prioritised action register as a deliverable
- Established clear next-step engagement scope

This is precisely the AIRL delivery model for UK Public Sector and Mid-Market. The insurance sector proof-of-concept validates the commercial pattern before the AIRL product is fully productised.

### 1.2 The Azure Assessment Parallel

Azure Well-Architected Reviews, CAF assessments, and Azure Migrate assessments follow the same pattern — but stop at the scoring layer. They produce:

- A pillar-by-pillar maturity score (Reliability, Security, Cost, Operations, Performance)
- A recommendations list ranked by impact
- No sprint plan, no delivery ownership, no PbD/SbD gate sequencing

**The gap between the Azure assessment output and an executable delivery programme is exactly the market space PFI-AIRL occupies.** The Snapshot Audit model proves clients will pay for that conversion. Azure Assessments are the industrialised, Microsoft-validated version of the discovery phase — and they run at scale across every Azure-committed organisation.

## 2. VE Skills Chain Analysis

### 2.1 VSOM Layer — Vision and Strategic Positioning

The existing PFI-AIRL vision positions the platform as:

> *"The definitive AI Readiness and Azure Intelligence Layer — accelerating idea-to-execution through ontology-driven assessment, ALZ-native architecture, and agentic AI augmentation."*

Azure Skills Builder and Azure Assessments map directly onto this vision as the **input tier** in a two-tier model:

```text
Tier 1 (Microsoft) → Azure Assessments + Skills Builder
                      Surface gaps, produce scores, identify capability deficits

Tier 2 (PFI-AIRL)  → AIRL Ontology Engine + Agentic Pipeline + PbD/SbD Gates
                      Convert scores to sprint roadmaps, bridge skills to readiness,
                      close gaps with Zero Trust ALZ + Privacy-Native Data Platform
```

**VSOM Strategic Implication:** The five PFI-AIRL strategy pillars are all accelerated when Microsoft assessment data is treated as structured input:

| Strategy Pillar | Azure Leverage |
|---|---|
| S1 · AI Readiness Assessment | WAR Security + WAR Reliability pillars feed directly into AIRL domains 1–5 |
| S2 · ALZ Accelerator | WAR findings map to ALZ blueprint selection — automate via AIRL ingest |
| S3 · Data & AI Platform | CAF Data & AI assessment outputs scope the Data Platform sprint |
| S4 · Velocity Framework | Skills Builder gap analysis auto-sequences sprint team composition requirements |
| S5 · PbD/SbD Foundations | **WAR has no PbD/SbD scoring** — this omission is PFI-AIRL's structural moat |

The moat deserves emphasis. Microsoft's Well-Architected Framework has five pillars. **None of them is Privacy by Design. None is Security by Design in the proactive, ISO 31700 / NCSC SbD sense.** Microsoft has a commercial incentive not to score clients on privacy maturity — it sells the data platform that processes the data. PFI-AIRL has no such conflict. The AIRL 9-domain model (including PbD Maturity 15% + SbD Maturity 15%) occupies territory Microsoft cannot credibly enter.

The most commercially significant pain is the **post-assessment void** — the period between receiving a Microsoft assessment report and knowing what to do next. This period typically lasts weeks to months, involves internal debate, and often results in no action or a consultant engagement that redoes the assessment under a different name.

PFI-AIRL, via the WAR/CAF ingest pipeline, eliminates the post-assessment void by converting the assessment output into an AIRL sprint plan on the same day. This is a differentiated pain reliever that no current UK market participant offers.

### 2.5 PMF Layer

#### Ideal Customer Profile — Sharpened by Azure Assessment Context

The existing ICPs (UK Public Sector and Mid-Market ≤£500M) are both **already inside Microsoft's assessment programmes**. Public sector organisations on Azure CSP agreements run WAR reviews as part of CSP governance cadence. Mid-market clients often receive CAF assessments as part of Azure migration projects facilitated by Microsoft partners.

This means PFI-AIRL does not need to generate the initial assessment demand — **Microsoft already creates it at scale.** The PMF question becomes: can AIRL reliably capture post-assessment clients at the moment of peak intent?

**The Post-Assessment Conversion Funnel:**

```text
Microsoft CSP/MPN Partner delivers WAR review
              ↓
Client receives score + recommendations — no sprint plan
              ↓
Post-Assessment Void begins (typically 4–12 weeks)
              ↓
PFI-AIRL engages — AIRL ingests WAR output
              ↓
Sprint plan delivered within 1 business day
              ↓
Velocity Framework activates — 30-day TTV sprint
              ↓
PbD/SbD gates validate at every stage transition
```

This is a **warm market entry at peak intent.** The client has already validated the problem (Microsoft told them they have gaps). PFI-AIRL arrives with a solution the client cannot get from Microsoft or from their existing partner.

#### Additional PMF Signal

**Post-Assessment AIRL Conversion Rate** — the percentage of AIRL engagements where the client had a prior Microsoft assessment within the preceding 12 months. If this signal is ≥50%, it validates the Microsoft ecosystem as the primary pipeline source and informs GTM prioritisation: focus on Microsoft CSP partners as a referral channel rather than direct market outreach.

## 3. Delivery Architecture — The AIRL Ingest Pipeline

### 3.1 WAR/CAF → AIRL Domain Mapping

The AIRL 9-domain model maps to WAR pillars as follows:

| Azure WAR Pillar | AIRL Domain | Mapping Type |
|---|---|---|
| Reliability | AIRL-D03: Infrastructure Readiness | Direct |
| Security | AIRL-D08: Security by Design Maturity | Partial — WAR Security scores technical controls; AIRL-D08 adds SbD proactive design |
| Cost Optimisation | AIRL-D05: Financial & Commercial Readiness | Partial — cost efficiency signals FinOps maturity |
| Operational Excellence | AIRL-D04: Process & Operational Maturity | Direct |
| Performance Efficiency | AIRL-D02: Technical Architecture Readiness | Direct |
| *(No WAR equivalent)* | **AIRL-D07: Privacy by Design Maturity** | **AIRL-only — structural gap in WAR** |
| *(No WAR equivalent)* | **AIRL-D09: AI Governance Readiness** | **AIRL-only — structural gap in WAR** |

The two AIRL-only domains — PbD Maturity and AI Governance — represent 30% of the AIRL composite score and are entirely invisible to WAR. This delta is the opening line of every post-WAR AIRL conversation.

### 3.2 Skills Builder → AIRL Domain Mapping

Azure Skills Builder identifies gaps at the individual role level. AIRL maps these to project readiness domains:

| Skills Builder Deficit | AIRL Domain Risk Signal | Sprint Implication |
|---|---|---|
| Azure AI Foundry skills absent | AIRL-D01: AI Capability Maturity — RED | Sprint requires AI foundations sprint as prerequisite |
| Azure Purview / data governance skills absent | AIRL-D07: PbD Maturity — AMBER | PII classification sprint added to ALZ Phase 1 |
| Azure Policy / Compliance skills absent | AIRL-D06: Compliance & Regulatory — RED | Compliance sprint precedes data platform sprint |
| Azure Sentinel / Defender skills absent | AIRL-D08: SbD Maturity — RED | Security baseline sprint mandatory before ALZ deploy |
| Azure AI Foundry + Responsible AI skills absent | AIRL-D09: AI Governance — RED | AI governance sprint gates AI model deployment |

This mapping converts a Skills Builder export from a learning & development report into a **sprint sequencing constraint** — a materially more useful output for a CIO or CDO audience.

### 3.3 Agentic Pipeline Architecture

The ingest pipeline extends the existing AIRL agentic infrastructure (N8N + Claude):

```text
Input Sources
├── WAR Export (JSON)     → Azure Portal WAR export
├── CAF Assessment        → CAF Maturity Model structured output
└── Skills Builder Export → Team capability gap report (CSV/JSON)
         ↓
AIRL Ingest Agent (N8N + Claude)
├── Parse → normalise to AIRL 9-domain schema
├── Score → apply AIRL domain weighting model
├── Gap-map → identify PbD/SbD deltas (WAR omissions)
└── Sequence → generate sprint dependency graph
         ↓
AIRL Ontology Layer
├── VP-ONT: map findings to customer jobs/pains/gains
├── RRR-ONT: map risks → requirements → results
├── KPI-ONT: attach metrics to sprint outcomes
└── BSC-ONT: map to BSC quadrant dashboard
         ↓
Output Artefacts
├── AIRL Scorecard (9 domains, composite score, PbD/SbD deltas)
├── Sprint Plan (Velocity Framework sequenced)
├── PbD/SbD Gate Schedule (5 gates, dates, owners)
└── DPIA Scope Register (auto-generated if PbD gaps detected)
```

**Target delivery time:** ≤4 hours from structured assessment input to complete AIRL sprint plan.

## 4. Partner Positioning — Microsoft Co-Sell

### 4.1 The Co-Sell Opportunity

Microsoft's MPN (Microsoft Partner Network) and CSP programme includes a post-assessment execution gap that Microsoft actively wants partners to fill. The Azure Expert MSP designation and Azure specialisations create a channel where Microsoft refers clients to partners after assessments surface complexity.

PFI-AIRL's positioning in this channel:

> *"Post-assessment AI execution partner — WAR/CAF-native, ALZ-accredited, PbD/SbD-certified delivery. Converts your Microsoft assessment score into an executable 30-day sprint plan on the same day."*

This positioning is:

- **Partner-safe** — additive to Microsoft's assessment tooling, not substitutive
- **Differentiated** — no other UK partner offers PbD/SbD-gated sprint execution
- **Measurable** — ALZ deploy in ≤21 days, DPIA in <2 hours, AIRL score in ≤5 days

### 4.2 OKR Alignment

This partner channel directly accelerates:

- **KR2.3:** 2 Gov framework agreements — co-sell agreements with Microsoft CSP partners provide the procurement pathway into public sector frameworks
- **KR1.2:** 3 framework endorsements — Microsoft endorsement of AIRL as a post-WAR execution tool carries sector credibility equivalent to an ICO or NCSC endorsement
- **KR2.2:** £2M ALZ ARR — co-sell referral pipeline from Microsoft WAR reviews is the highest-volume lead source available in the UK Azure mid-market

### 4.3 Provenance Transparency

For UK public sector procurement, clients need to demonstrate that their assessment basis is auditable. The AIRL scoring model should declare source provenance:

| Score Component | Source | Audit Trail |
|---|---|---|
| WAR-derived domains | Microsoft WAR JSON export | Date, assessor, WAR version |
| AIRL-native domains (PbD/SbD, AI Governance) | AIRL proprietary scoring | Methodology doc, ontology version |
| Skills-derived readiness signal | Azure Skills Builder export | Date, scope, role coverage |
| Composite AIRL Score | AIRL weighted model | Weighting version, ontology ref |

This provenance model supports ICO, NHS DSP Toolkit, and Cyber Essentials Plus evidence requirements without additional documentation effort.
