## 5. Unified Registry Architecture

**Source:** [PFC-REG-BRIEF-Unified-Registry-Skills-Architecture-v1.0.0.md](https://github.com/ajrmooreuk/Azlan-EA-AAA/blob/main/PBS/STRATEGY/PFC-REG-BRIEF-Unified-Registry-Skills-Architecture-v1.0.0.md)

### Core Principles

- **ONE registry** for ALL artifact types — ontologies, skills, plugins, agents, tokens, schemas, PRDs
- **Quasi-OO cascade:** Core (base) → Instance (subclass) → Product (specialisation) → Client (final customisation)
- **Per-scope states:** ON (inherit) | ON (customised) | OFF (disabled)
- **Skills are registry artifacts** — same pattern as ontologies, design tokens, agent prompts

### Cascade Pattern

```
PFC (Core)          → base definitions, all ontologies, core skills
  └─ PFI (Instance) → inherit + override (e.g. BAIV, W4M, AIRL)
      └─ Product    → specialise (e.g. MarTech, Supply-Chain)
          └─ Client → final customisation per end-customer
```

### Proposed Infrastructure

- **Database:** Supabase `pfc_registry` — single table for all artifact types
- **CI/CD:** Single `registry-ci.yml` validates ALL artifacts (JSON-LD, Schema.org, OAA, dependency graph, override compatibility)
- **File system:** 1:1 mapping — registry is logical layer, file system is physical implementation

## 6. PE-ONT v4.1.0 — New Entities for Skills & Process Paths

**Source:** [process-engineering-v4.1.0-oaa-v7.json](https://github.com/ajrmooreuk/Azlan-EA-AAA/blob/main/PBS/ONTOLOGIES/ontology-library/PE-Series/PE-ONT/process-engineering-v4.1.0-oaa-v7.json)

| Entity | Purpose |
|---|---|
| `pe:Skill` | Atomic callable unit — skillId, version, typed inputs/outputs, provider enum, owningOntology |
| `pe:Plugin` | Extension bundle — cascadeTier (PFC/PFI/Product/App), configSchema, entryPoint. Must expose ≥1 Skill (BR-016) |
| `pe:ProcessPath` | Directed graph of process steps — no fixed start, pathType, scope, seriesRefs |
| `pe:PathStep` | Graph node — stepType, entryPoint flag, conditionExpr, refs Process OR Skill (never both, BR-013) |
| `pe:PathLink` | Directed edge — bindingStrength (mandatory/recommended/optional/conditional), linkType |

### Key Design Decisions

- **DD-001:** ProcessPath is a directed graph, NOT a sequence — no fixed start node, entry determined by context
- **DD-002:** Skills are cross-cutting — any process in any series can invoke any skill
- **DD-003:** `bindingStrength` on PathLink is the core flexibility mechanism — mandatory for strict chains, optional for forks
